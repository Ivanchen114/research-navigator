# 教學網站 research-navigator 優化計畫

> 撰寫於 2026-04-27，基於對 51 個頁面、29,447 行程式碼（pages + games + components）的稽核與教學設計評估。
> 本文件作為**長期維護地圖**，不是立即執行清單。每項都標明「何時動、為何動、風險級別」。

---

## Context — 為什麼提這份計畫

這套網站以「研究方法教學職人手作」的水準完成，教學設計、敘事編排、AI 協作論、Prompt 設計每層都頂級。但作為一個**會繼續迭代的活產品**，目前有幾個面向是隱性債務：

1. **單點失敗風險**：18 週設計理路只在程式碼與你的腦中，沒有任何文件講清楚「為什麼 W3 是診斷、W7 是分流、W11 是倫理」這個整體骨架。如果未來要交接、培訓助教、或你自己一年後回頭改，會痛。
2. **技術債**：CSS 變數散落 17 個頁面、單頁 1000+ 行、Tailwind 與 inline style 並存。能跑但難維護。
3. **教學內容盲點**：兩個維度沒被涵蓋（研究失敗常態、研究品質質化判斷），但這需要你決定要不要納入。
4. **資料黑盒**：學生在哪裡卡住、哪個 Prompt 沒人複製、哪個遊戲沒人玩完—完全沒紀錄。

每項優化的 ROI 不同。**沒有一條建議是「必須做」**，全部都是判斷。

## 🔴 P1 — 寫 18 週設計理路 design doc（現在就該做）

### 為什麼

如果你今天忽然不能繼續維護這個網站（或你自己一年後回頭看），**沒有任何一份文件能讓人理解「這 18 週為什麼這樣設計」**。
你會看到 51 個頁面、19 個遊戲、AIRED 框架、人機協作框架、兩層判斷架構、Glow & Grow 形成性回饋—但這些拼圖如何拼成完整的研究方法課程理路，**只在你腦中**。

這是單點失敗風險最大的一塊。一份文件能：
- 讓未來接手的老師/助教 30 分鐘內掌握全貌
- 讓你自己之後修改某週時，能快速回憶「為什麼當初這樣設計」
- 對外發表（教師研習、論文）時直接有現成材料

### 建議結構

`docs/COURSE_DESIGN_RATIONALE.md`（在 research-navigator 根目錄底下開個 `docs/`）：

```markdown
# 研究方法課程設計理路（18 週）

## 一、課程哲學（你的設計原則）
- 為什麼把研究方法包成「探案敘事」而不是傳統章節
- AI 協作為什麼是核心而不是輔助
- 「碰壁體驗 → 急救心法」的教學論依據（productive failure）
- AIRED 五要素為何這樣切

## 二、18 週骨架圖
- 五大階段：探索 → 診斷 → 分流 → 設計 → 執行 → 整合
- 每階段對應的方法學概念與認知層次（Bloom）
- 階段間的「銜接點」設計（W2→W3 怎麼過渡、W7→W8 怎麼從個人到組隊）

## 三、各週設計筆記（每週 1-2 段）
W1：為什麼從 RED 公約開始
W3：診斷八病的選題依據（為什麼是這 8 種）
W7：兩層架構的來源（哪本書、哪位學者）
W11：倫理四問為什麼簡化成這四個
...

## 四、評量設計（接續批改系統）
- Rule + LLM 雙軌的設計理由
- Glow & Grow 的形成性回饋論基礎
- Tier 1/2/3 分流的邏輯

## 五、技術選型備忘
- 為什麼 React + Vite 而不是 Next/Astro
- 為什麼大量 inline style（離線/單檔可移植考量？）
- 為什麼每週一頁而不是 template 化

## 六、後續開發優先級（指向本文件）
```

### 動作

1. 你口述 + 我整理，分 3-4 次對話就寫完（每次聚焦一個章節）
2. 或者你自己寫，遇到要點程式碼證據時讓我去翻

## 🟠 P2.1 — CSS 設計系統統一（學期結束做）

### 現況證據

```jsx
// Home.jsx:32-50（每頁都來一份）
<style dangerouslySetInnerHTML={{
  __html: `
    :root {
      --ink: #1a1a2e;
      --ink-mid: #4a4a6a;
      --paper: #f7f5f0;
      --accent: #2d5be3;
      --gold: #c9a84c;
      ...
    }
  `
}} />
```

W11.css、W12.css…W17.css 各自寫一份。Tailwind 4 設定也在跑。**三套系統並存。**

### 痛點觸發條件

- **想全站換配色** → 要改 17 個檔案
- **想新增一個共用色變數** → 不知道現有變數有哪些，因為散落各處
- **想統一字級或間距系統** → 每頁都要重看一遍

### 建議做法

1. 開一個 `src/styles/design-tokens.css`，把所有 `--ink, --paper, --accent, --gold...` 集中
2. 在 `src/main.jsx` import 一次（全站共享）
3. 各頁的 `<style dangerouslySetInnerHTML>` 改成只寫該頁特有的樣式（移除 `:root` 區塊）
4. WX.css 內凡是「Tailwind 能做的」遷移到 className（如 padding、margin、color）；保留只有「Tailwind 不容易做的」（如複雜的 keyframes 動畫、grid template）
5. 估計能砍 30-40% 的 CSS 行數

### 風險

- 視覺迴歸：每頁要逐頁檢視
- **絕對不要學期中做**，學生正在用網站

### 驗證

- `npm run build` 仍正常
- 逐頁手動瀏覽（Home → W1 → W3 → W7 → ... → W17 → 各遊戲），確認視覺無變化
- 把舊版本截圖（每頁一張），對比新版本

## 🟠 P2.2 — 大頁面拆分（學期結束做）

### 現況證據

```
W9Page.jsx          1276 行
Wizard.jsx          1213 行
TeamFormation.jsx    995 行
ClinicPage.jsx       897 行
DataAnalysisStation  800 行
LiteratureReview     790 行
Portfolio.jsx        745 行
PromptLab.jsx        718 行
W11Page.jsx          704 行
```

每個檔案內都有：常數定義 + 內部元件 + 主元件 + 副作用 + Hooks。

### 痛點觸發條件

- 修改某個 Step 的 prompt 時，要 scroll 過 800 行才找到
- 內部元件（如 W11 的 `CopyablePrompt`）其實多頁通用但散落各處
- 開發新週次時，沒有清楚的「拆分範本」可參考

### 建議做法（保守，不破壞現有結構）

每個 1000+ 行的頁面拆成：

```
src/pages/W9/
  index.jsx              // 主頁面（< 200 行，只組合）
  data.js                // 常數、配置（如 PILOT_INSTRUCTIONS）
  prompts.js             // AI Prompts 集中
  steps.jsx              // Step 元件（如果有的話）
```

或更輕量：把 inline 元件抽到 `src/components/ui/` 共用（特別是 `CopyablePrompt`、各種 Step 卡片）。

### 風險

- 動到全站 4-9 頁
- 內部元件重新 import path，要小心
- **絕對不要學期中做**

### 驗證

- 逐頁手動瀏覽
- 確認 export field key 一致（學生資料按 key 存在 localStorage，改變會丟資料）

## 🟡 P3.1 — 加入「研究失敗是常態」framing（教學設計決策）

### 為什麼

從 lessonMaps 看，目前所有週次都假設學生會「走完完整研究歷程」：
- W11 工具修正 + 倫理審查
- W12-W13 執行
- W14 結論
- W15-W16 發表

**但真實研究 60% 以上的學生會碰到中途要轉向**：
- Pilot Test 後發現工具完全要重做
- 樣本招募完全失敗（同學不填問卷、訪談對象消失）
- 實驗組設備故障/時間不夠
- 文獻分析發現題目其實已經被做過

目前沒看到任何學習單在處理「研究失敗 / 中途轉向」這個議題。學生遇到時可能會：
- 自責（「是我能力不好」）
- 隱瞞（「假裝有結果」）
- 放棄（「擺爛交差」）

但研究失敗其實是 **可以教的能力**——叫做 *研究調整*（research pivot）：
- 認識到失敗的訊號（早期還是晚期？）
- 評估是「修補」還是「轉向」
- 重新設計 Plan B 並縮小範圍
- 把失敗紀錄寫進最終報告（這是學術誠信）

### 建議的內容形式

選項 A：**新增 W11.5 / W12.5「研究調整工作坊」**（半節課），用一張學習單帶過：

```
1. 我預期會發生 ___（執行前寫）
2. 實際發生了 ___（執行中寫）
3. 兩者差距是因為 ___
4. 我選擇：(a) 修補原計畫 (b) 轉向 Plan B (c) 重新縮小題目
5. 新計畫是 ___
6. 從這次經驗我學到 ___
```

選項 B：**在 W12 / W13 既有頁面加一個「卡住了？」按鈕**，點開引導學生填上面那六題、提供 AI 諮詢 Prompt：

```
我的研究計畫是 [貼上]，目前在 [貼上實際狀況]，
請從研究方法的角度幫我判斷：
1. 這是早期失敗訊號還是晚期？
2. 修補的成本與轉向的成本各是什麼？
3. 三個我可以採取的具體行動是什麼？
```

### 風險

- 設計決策：要不要把「失敗」拉到課程明面上？有些老師會擔心給學生消極暗示
- 但學界共識：教研究的人**不教失敗就是不誠實**
- **由你決定**

## 🟡 P3.2 — 加入「研究品質質化判斷」rubric（教學設計決策）

### 為什麼

從 W11 倫理四問、Pilot Test 設計、Step 引擎結構看，大量檢查項是 binary（有沒有寫、有沒有勾）。但「研究品質的真實判斷」更深的問題沒被處理：

- **操作型定義**：「快樂」要怎麼測？問「你快樂嗎」就是測快樂嗎？
- **建構效度**：「拖延」量表的題目真的測到拖延嗎？還是測到時間管理？
- **測量信度**：你的觀察類別兩個人會分到同一類嗎？
- **抽樣偏誤**：問卷只在自己班級發，能推論「全校」嗎？
- **訪談引導性**：「你應該很喜歡 IG 對吧？」這種問題會引導答案

這些是研究方法的「進階品質判斷」。批改系統那邊我也提過同樣問題。

### 建議

W14（研究結論）或 W15（發表前）加一張「研究品質自評表」：

| 維度 | 滿分準則 | 自評（◎好 ○中 △待改 ✗未做） | 改善計畫 |
|------|---------|------------------------|---------|
| 操作型定義 | 我的核心變項有清楚的測量方式，外人能照著測 | | |
| 樣本代表性 | 我的樣本能代表哪個母體？哪些不能？ | | |
| 信度檢核 | 我有做過某種一致性檢驗（複測、跨評者一致、α 係數）| | |
| 訪談 / 問卷引導性 | 我有讓他人檢視題目是否誘導 | | |
| 推論限制 | 我清楚說明本研究的結論「不適用於」哪些情境 | | |

也提供 AI Prompt：「我的研究是 [貼上]，請從研究方法品質角度幫我找出三個最可能被質疑的點」。

### 風險

- 對高一/高二學生可能太抽象
- 但你說過要追求「專業高中教師」水準，這就是分水嶺
- **由你決定**

## ⚪ P4 — 加入 analytics（暑假可選）

### 為什麼

學生用了一整學期網站，你目前**完全不知道**：
- 哪個頁面 bounce rate 最高（學生打開馬上關）
- 哪個 Prompt 沒人複製
- 哪個遊戲沒人玩完
- 哪個 Step 卡最久
- AIRED 敘事填寫率

下次設計改版要改什麼，只能憑直覺。

### 建議

**最輕量做法**：用免費版 Plausible 或 Umami（不用 cookie banner、台灣不用考慮 GDPR），加 5 行 script tag 進 `index.html` 就好：

```html
<script defer data-domain="research-navigator.vercel.app"
        src="https://plausible.io/js/script.js"></script>
```

進階：在 Step 完成、Prompt 複製、遊戲過關時觸發 custom event：
```js
window.plausible('prompt-copy', { props: { week: 'w11', promptId: 'consent-ai' } });
```

### 風險

- **隱私倫理**：學生資料就算匿名也要老師告知 + 學校知情。
- 你的課堂規模可能不大，data 不一定夠統計顯著
- 但有資料總比沒資料強

---

## 不在這份計畫裡的東西（為什麼）

- **加更多遊戲 / 章節**：你已經 19 個了，不缺
- **改造成 multi-tenant（多老師共用）**：那是商業化決策不是優化
- **重寫成 Next.js / Astro**：沒有真的痛點觸發，純粹技術潔癖
- **TypeScript 遷移**：JS 已經能跑，遷移成本對個人開發者太大

## 建議順序

1. **這個月**：跟我配合寫 P1（design doc）
2. **學期末（七月）**：P2.1 + P2.2（技術重構）

如果只能做一件：**P1**。技術債未來都還能補，但「設計理路文件化」越拖越記不住。
