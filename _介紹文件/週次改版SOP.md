## 核心原則：方向寫網頁，故事留老師講

學生在課堂上不是讀講義，是做任務。網頁要幫學生隨時回答三句話：

> 1. **我現在在哪一步？**（StepEngine 已處理）
> 2. **我現在要做什麼？**（StepBriefing 處理）
> 3. **做完要去哪裡？**（StepBriefing「完成後」+ 下一步按鈕）

網頁不承擔所有說明——細節留給講稿。

## 一、改版工作流程（9 步）

每個週次改版照這個順序走：

### 步驟 1：盤點現有資料夾
- 列出 `00新教案/W{N}/` 下所有檔案
- 列出 `學習單批改系統/W{N}_學習單/` 內容
- 確認哪些檔案要動、哪些要保留

### 步驟 2：搬舊檔到 _archive（先疊加，不覆蓋）
```bash
mkdir -p "_archive/YYYY-MM-DD_pre_X改版/"
mv W{N}_教師逐字講稿.md W{N}_完整教案.md W{N}_酷AI教材文本.md "_archive/..."
```
依記憶慣例：**改大版前必須先疊加歸檔**，不要直接覆蓋。

### 步驟 3：讀現網頁完整流程
- Read `src/pages/W{N}Page.jsx` 完整內容
- 不能憑印象——「**寫教案前先讀真實程式碼**」（記憶教訓）

### 步驟 4：找化石矛盾
依下方「化石掃描清單」逐項檢查。

### 步驟 5：寫講稿（對齊現網頁）

### 步驟 6：套 StepBriefing
每個 Step 一張，依下方「StepBriefing 字數標準」寫。

### 步驟 7：修化石（網頁 + lessonMaps.js + portfolioRegistry.js + 講稿）

### 步驟 8：build 驗證
```bash
cd /Users/Ivan/Documents/test/研究方法/00新教案/code/research-navigator
rm -f dist/.DS_Store dist/assets/.DS_Store
npm run build
```
依專案規則：遇到 EPERM `.DS_Store` 錯誤先刪 `dist/.DS_Store`。

### 步驟 9：說「已改」前必須 grep 驗證 0 殘留
依記憶教訓：「**說「已改」前必須 grep 全站驗證**」——不能憑記憶。

### 步驟 10：接收外部審查時，先 grep 對方提的每一條
如果有別人（同事、其他 AI、GPT 等）審查網頁/講稿並列出「沒修好的地方」——**不要直接動工修**。
先做這件事：
- 對方提的每條「還沒修」，**用 grep 驗證當前實況**
- 因為對方可能在看：舊版檔案、Git 較早 commit、瀏覽器快取、自己的記憶
- W1 改版時遇過：GPT 列 9 條「沒修好」，grep 後發現 8 條早就改完——他在憑舊版判斷
- 這跟「自己說已改前要 grep」是同一個原則的反面：**接收審查也要 grep**
- 否則會被「審查疲勞」拖入循環修補，浪費時間且把對的東西改錯

**反向觀察**：如果要請外部 AI 審查，**提供最新檔案連結**而非貼舊版片段——降低誤判機率。

## 二、StepBriefing 字數標準（鐵律）

每個 Step 開頭固定加一張 `<StepBriefing task done />`。

### 字數鐵律
| 欄位 | 字數上限 | 範例 |
|---|---|---|
| **本步任務** | **30 字內** | 「閱讀 7 份回答，找出 AI 寫的那份，寫下理由。」 |
| **完成後** | **20 字內** | 「按「揭曉真相」——要按 10 下才解鎖。」 |

### 設計鐵律
- **task 是祈使句**，動詞開頭（寫、讀、找、選、按、貼）
- **done 是下一步動作**，不是理念或抒情
- **不要重複內文**已經講過的細節
- 超過字數 → 砍掉「為什麼」「怎麼樣」，只留「做什麼」

### 反例（不要這樣寫）
```
❌ task: 「閱讀 7 份回答，判斷哪一份最像 AI 寫的，並寫下至少 20 字理由。完成後請按下揭曉真相按鈕，並觀察自己原本判斷和答案之間的差距。」(60+ 字)
```

### 正例
```
✅ task: 「閱讀 7 份回答，找出 AI 寫的那份，寫下理由。」(22 字)
✅ done: 「按「揭曉真相」——要按 10 下才解鎖。」(18 字)
```

### 視覺位置
```jsx
<div>
    <div>PART X · 副標</div>
    <h3>標題</h3>
    <p>一段簡介說明...</p>
</div>

<StepBriefing task="..." done="..." />   ← 這裡

<ThinkRecord ... />   ← 實際操作區
```

## 三、化石掃描清單

每次改版前用這份 checklist 全頁掃。

### A. 前後指涉化石

- [ ] **TaskCard tasks**：三項是否對應實際 Step 動作？

- [ ] **W2 預告區**：「你要帶來的東西」指涉的 Step 編號正確嗎？

### C. 跨檔同步化石
- [ ] **lessonMaps.js W{N}Data**：metaCards / coreConcepts / periods stages / summaries 是否對齊網頁 5 step？
- [ ] **portfolioRegistry.js W{N} fields**：EXPORT_FIELDS 是否同步？砍掉的 dataKey 是否同步砍掉？
- [ ] **講稿節奏總表**：每個 Step 描述是否對齊網頁？

### D. AI-RED 字母對應化石
- [ ] **不要為了湊 A-I-R-E-D 把每個研究步驟硬塞字母**
- [ ] AI-RED 是「**使用 AI 時的紀錄框架**」，不是「研究流程五步驟」
- [ ] **只在真正用 AI 的步驟標 AI-RED 字母**：
  - A（Ascribe）：說明用了哪個 AI、做什麼
  - I（Inquire）：Prompt 提問

  - D（Document）：記錄最後決策
- [ ] 「確認研究問題」「現場收原始資料」不是 AI-RED 任何一個字母——是研究階段的人類判斷

## 四、Framing 慣例（重要：跨週共用）

### 1. 不能外包，不是 AI 做不到
- ❌ 「**三件 AI 做不到的事**」（脆弱，AI 進步會破）
- ✅ 「**這學期你不能外包的三件事**」（責任分配，永遠成立）

三件事內涵：
| | 內容 |
|---|---|
| **品味（Taste）** | AI 可以幫你列問題，但**不能替你決定**哪個問題值得研究 |
| **接觸（Touch）** | AI 可以整理資料，但**不能替你親自**觀察、訪問、蒐集證據 |
| **判斷（Judgment）** | AI 會胡說八道、產生錯誤偏誤、把資料整理得太漂亮；**你要回頭檢查** |

### 2. 捍衛真實，不是防弊
- ❌ 「AI-RED 是給老師抓的工具」
- ✅ 「AI-RED 是給自己看的——讓你自己分得清，哪些是 AI 幫的，哪些是你判斷後決定留下的」

### 3. 軍階保留，但加緩衝
- 「**軍階紀錄與研究技能足跡**」（保留 R.I.B. 主題詞）
- **必加緩衝句**：「**不算成績、也不是排名**」

### 4. 18 週中心問題引線
每週 HeroBlock 都有一個中心問題（已全部裝載）。回扣 W1 的「**一段看起來像研究的文字，憑什麼相信？**」——每週用不同角度回答。

## 七、結構性判斷反向挑戰鐵律（W9 教訓）

審查或改版時，**凡涉及章節編號、週次來源、dataKey、路由檔名、跨檔依賴**這類結構性判斷，**不可憑印象判斷或反駁外部審查**——必須先 grep 或查當週權威表，再下判斷。

### 為什麼這條鐵律存在

2026-05-13 W8 SOP 收尾時用「全站鎖死第二章=文獻」反向挑戰 GPT v2 建議「第二章改文獻回顧章節」，但實際**沒 grep W9 章節對應表權威版**——導致誤判把文獻探討段落歸進第二章（實際是第三章）。一輪「不改」的決定讓錯誤跨 12 處 5 檔擴散。

錯誤性質：**權威來源未確認 → 錯誤章節假設 → 反向挑戰 GPT 用錯前提 → 錯誤同步到多檔**。看起來很有邏輯，但其實是在錯誤地圖上走得很認真。

### 權威表查找順序（從高到低）

1. **當週頁面自己的章節對應表 / 常數陣列**（如 W9Page line 1428-1437 的 13 章 map）
2. lessonMaps.js 該週 Data
3. portfolioRegistry.js
4. 逐字稿
5. 其他週次頁面殘留文字（最低權重）

### 反向挑戰外部審查時的硬規則

- 「我已驗證」「全站鎖死」「跨檔依賴」這類詞出現時 → **當場 grep 出證據貼在判斷裡**，不能靠記憶
- 跨檔斷言（例「W6/W8/W9/W10/ToolRefinement 全鎖死第二章」）必須 grep 出該詞在每個檔案的實際出現位置
- 如果 grep 結果跟記憶不符，**信 grep、不信記憶**

### grep 範圍盲點（W9 第二次教訓）

掃化石時 grep 範圍必須包含：

1. **`/src`** 內網頁 + lessonMaps.js + portfolioRegistry.js（這個自然會做）
2. **`/00新教案/W{N}/`** 內教案 .md 檔（**容易漏**——逐字稿/完整教案/酷 AI 教材文本）
3. **`/_介紹文件/`** 內 SOP 與課程說明檔

排除：`_trash/` 內舊 build 快照（用 `--glob !**/_trash/**`）

### 抓「症狀」 vs 抓「前提」

grep pattern 設計要分兩層：

| 層 | 抓的內容 | 範例 |
|---|---|---|
| **症狀層** | 直接違規字串 | 「第二章文獻」「W8 老師回饋」 |
| **前提層** | 違規前提假設 | 「W7-W8 計畫書」（前提是 W7-W8 在寫計畫書，但實際 W9 才是計畫書地基週） |

W9 第二次教訓：只抓「第二章 vs 文獻」這種症狀，沒抓到「W7-W8 計畫書」這種前提假設。前提錯誤比症狀更隱蔽——學生看了會以為「W7-W8 是計畫書週」，整個學習地圖會歪。

**對策**：每次修補章節/週次跨週鏈時，**多 grep 一輪「週次+用途」組合**（如「W7-W8 計畫書」「W5 工具」「W8 對象」）驗證前提是否符合現實課程結構。

### 「鎖死」這個詞要謹慎使用

技術上沒有任何 lock 機制——只有「多檔重複文案要一起維護」的人為共識。改詞前先問：

- 是程式技術鎖（import 常數、dataKey）？→ 真鎖，改要全站同步
- 是學術慣例鎖（如「研究計畫書第二章 = 操作型定義」是教科書結構）？→ 可拆鎖但成本高
- 是純文案重複？→ 隨時可改，但要 grep 全部出現位置
