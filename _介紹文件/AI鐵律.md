# AI 鐵律

> 給每次新對話的 Claude（或其他 AI 協作者）。動工前先讀完。
>
> 這份是「行為鐵律」，跟 `週次改版SOP.md`（流程鐵律）分工——SOP 寫「做什麼順序」，這份寫「動手時不要踩什麼坑」。

## 修改／審查／化石清理時

### 1. 兩層 grep（症狀層 + 前提層）
不只 grep 舊詞字串（症狀層），還要 grep 舊邏輯假設（前提層）。

**例**：W11 改命名「跨班 Pilot」→「跨方法 Pilot」
- 症狀層 grep：`跨班`
- 前提層 grep：W10 預告 W11 時用的「研究執行」、舊版「執行週 I」定位

**動手前先報**：「我準備 grep：[關鍵字列表]」，等使用者點頭再執行。這個動作會強迫你把不確定外顯化——說不出具體關鍵字時，那個遲疑本身就是「你還沒想清楚」的訊號。

### 2. 「預設不會錯的詞」才是真盲點
你會主動 grep 的詞，是你預期會錯的詞；但真盲點是你**預設不會錯的詞**。

修改時，要刻意把「我覺得這詞應該沒問題」的詞列入 grep 範圍。例：W11 改「本組工具書」→「本組工具設計書」時，學生口頭講「工具書」三個字就會混淆，「本組」前綴會被當成裝飾——這就是預設不會錯的詞。

### 3. 跨週改名要 grep 鄰週舊詞
動 W{N} 改某個用語後，要回頭 grep W{N-1}、W{N+1} 還有沒有舊詞殘留。

**例**：W11 改「跨方法 Pilot」後，要 grep W10 還有沒有「跨班 Pilot」（W10 在預告 W11 時用過舊詞）；W11 改「本組工具設計書」後，要看 W12 評分基準描述。

### 4. 外部審查清單先 grep 驗證命中率
GPT、其他 AI、人工審查給的修正清單，**先 grep 驗證每一條是不是真的還在**，不全信。常見原因：審查者看的是 iCloud Drive 同步前的舊版，或記憶中的版本。

實測過命中率多次落在 1/5 至 5/5 之間——不 grep 就動，就會誤改已修好的地方。

### 5. 說「已改」前必須 grep 全站確認 0 殘留
不能憑任務描述或記憶說「全部改完」。要實際 grep 確認 0 殘留——含教案 .md、lessonMaps、CSS 註解、_archive 外的所有引用。

### 6. 改動前先讀前後週脈絡
動 W{N} 前先 grep 前後週的 dataKey 鏈、主題定位、既有 state，否則容易產生不連貫。

## 任何時候

### 7. 不憑印象——從程式碼抓原文
命名、結構、dataKey、路由檔名、章節對應，要從程式碼／權威文件抓原文，不能憑印象虛構。

**陷阱地圖**：
- 路由 vs 檔名錯位（W7=W50Page.jsx、W8=LiteratureReview.jsx、W12-W17 檔名比路由少 1）
- 章節對應（計畫書第二章=操作型定義、第三章=文獻回顧——非「第二章=文獻」）
- dataKey 跨檔斷鏈（W6 寫 `w6-route`，W9 讀要 fallback 含 `w8-route`）

涉及這些不能憑印象——先 grep 或讀權威文件。

### 8. 設計存在 ≠ 行為可預期
看到某個機制／檔案／工具「設計上存在」，不代表它每次都觸發、每次都注入。

**例**：MEMORY.md 系統設計上存在，但實測有些 session 沒被注入到 Claude 的系統提示。所以不能依賴「Claude 自動會記得」——要主動把資訊寫進每次必讀的檔案。

**對應行為**：不確定就說不確定，不拿模糊敘述硬撐（「應該」「會」「機制是」這種沒驗證的肯定句要避免）。

### 9. build 用 dist-tmp 繞 iCloud 卡點
```bash
npx vite build --outDir dist-tmp --emptyOutDir
# build 完事後刪
rm -rf dist-tmp
```

**原因**：iCloud Drive 在 dist/ 留下 inode=65535 的同步殼目錄（`echo 2/`、`phantom 2/`、`w16 2/` 那種命名），會卡 vite 清 dist。用 dist-tmp 繞過，事後刪掉。

不要試圖在 sandbox 內手動 `rm` 那些目錄——sandbox 端的 rm 處理不了 iCloud 佔位符。要清就用 macOS Finder 手動清。

## 元認知：為什麼需要這份鐵律

**單人寫單人審天然偏弱**——你（建構者）在寫的時候腦中跑的是「下一段該寫什麼」（generative），審查者讀的時候是「現在這句對不對」（critical）。兩種注意力剛好錯開，所以建構者寫順了就放行，審查者卻能逐句抓矛盾。

這就是為什麼：
- GPT 審查能抓到我自己抓不到的字串（注意力方向相反）
- 我的 grep 清單 = 我預期會錯的詞 = 我的盲點清單（自己想不到自己的盲點）
- 「整體流程對」會掩蓋「細節矛盾」（建構者看大方向、審查者看每個字）

鐵律 1-6 是治本招——把建構者的「我以為我懂」強制變成「我說出我要查什麼」，自然就會卡在不確定的訊號上。
